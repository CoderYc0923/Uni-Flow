# 它解决什么

用一个真实场景串起来：**记账 / 客服意图分流**。

## 业务场景

用户发来一句话：

- 「午饭 32」→ 应记一笔流水  
- 「今天天气怎么样」→ 应闲聊，不要误记账  

你要的是：**稳定分流 + 各干各的 + 可测可加规则**。

## 没有统一编排时（乱）

常见两种做法都会痛：

1. **一个大 Prompt** 又判意图又记账又闲聊 → 难测、难加「记账必须走工具」这类硬规则  
2. **业务代码里手写调度**（`if route === 'record' …`、自己 for 循环排多个 Agent）→ 每个项目再发明一套「排班」，和模型代码缠在一起  

## 有 Uni-Flow 时（清）

```text
用户话
  └─ Router Unit（只负责判意图，产出 record / general）
        ├─ record  → 记账 Unit（写流水 / 调领域工具）
        └─ general → 闲聊 Unit
```

| 谁 | 干什么 | 放哪 |
|----|--------|------|
| Router / 顺序 / 预算 | 「怎么排班」 | Workflow YAML（声明） |
| 记账解析、写库、闲聊 Prompt | 「怎么干活」 | `uses` 插件（代码） |
| 引擎 | 按 YAML 调度 Unit，跑统一管线 | `uni-flow` 依赖 |

## 若你只记住一件事

**Uni-Flow 管「多 Agent 怎么编排」；不管你选哪个模型、怎么写业务 SQL。**
